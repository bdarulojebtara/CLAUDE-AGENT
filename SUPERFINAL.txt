  POR QUE DRIZZLE E NÃO PRISMA?
  Prisma gera um engine binário em Rust que precisa de `prisma generate` antes de cada deploy
  e causa problemas em servidores compartilhados brasileiros. Drizzle é JavaScript puro —
  instala com `npm install` e funciona imediatamente.

  REDIS OPCIONAL: se Redis não estiver instalado, o sistema detecta e usa cache em memória
  automaticamente. Em produção com Redis, sessões persistem entre reinicializações.

    - Após concluir: SETUP_COMPLETED=true, /install fica bloqueado

  FLUXO DEMO → PRODUÇÃO:
    1. Desenvolve e testa com DEMO_MODE=true no localhost
    2. No servidor: define DEMO_MODE=false no .env
    3. Acessa [dominio]/install no navegador
    4. Wizard: testa MySQL → dados da empresa → Admin → instala
    5. Sistema abre limpo, pronto para uso real

  BLOCO 07 — REGRAS DE NEGÓCIO INVIOLÁVEIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  REGRA 1 — SALDO REAL ANTI-FURO
    estoque_disponivel = estoque_atual - SUM(itens_op.quantidade_plan WHERE status='RESERVADO')
    Verificar NO BACKEND antes de confirmar venda ou abrir OP.
    Se insuficiente: HTTP 409 com detalhes de qual produto faltou.
    Também exibir visualmente em tempo real no frontend (TanStack Query polling ou WebSocket).

  REGRA 2 — CICLO DE MATERIAL NA FÁBRICA
    Ao abrir OP:    status = RESERVADO   → some do "disponível p/ venda"
    Ao iniciar OP:  status = EM_PRODUCAO → material saiu da prateleira
    Ao concluir OP: status = CONSUMIDO   → movimentacao CONSUMO_OP
    Ao cancelar OP: status = DEVOLVIDO   → movimentacao DEVOLUCAO_OP

  REGRA 3 — TRAVA DE FERRAMENTA
    Backend REJEITA saída de ferramenta nova se funcionário já tem ferramenta
    da MESMA CATEGORIA com status EM_USO.
    Exceção: perm_ferramenta_override com motivo obrigatório, gravado no log.

  REGRA 4 — SIGILO DE CUSTO ABSOLUTO
    Serializer do backend filtra preco_custo:
    if (!req.user.permissoes.includes('perm_produto_ver_custo')) {
      delete produto.preco_custo; delete produto.margem;
    }
    NUNCA expor em itens_venda para perfis sem perm_venda_ver_custo.
    Frontend TypeScript: tipo Produto tem preco_custo?: number (opcional) para refletir isso.

  REGRA 5 — SEPARAÇÃO CONDICIONADA
    Toda movimentacao_estoque de saída DEVE ter referencia_id e referencia_tipo.
    Backend recusa sem origem. Exceção: ajuste com perm_produto_ajustar_estoque + motivo.

  REGRA 6 — CANCELAMENTO COM SENHA ADMIN
    PUT /vendas/:id/cancelar exige body.senha_admin.
    Backend valida com bcrypt. Reverte movimentacoes_estoque.
    Registra: cancelada_por, motivo_cancel, auditoria, IP.

  REGRA 7 — ALERTAS ANTI-SEDEX (PROATIVOS)
    node-cron verifica estoque a cada 5 minutos.
    SE estoque_disponivel == 0              → 🔴 ESTOQUE_ZERO
    SE estoque_disponivel <= estoque_minimo → 🟠 ESTOQUE_CRITICO
    Anti-duplicata: nunca criar notificação se já existe não lida do mesmo tipo+produto.

  REGRA 8 — HISTÓRICO DE PREÇOS APPEND-ONLY
    Toda alteração em preco_custo ou preco_venda → inserir em historico_precos.
    Nas vendas: salvar SNAPSHOT (preco_unit e preco_custo em itens_venda).

  REGRA 9 — RETALHOS DE ALUMÍNIO
    Corte com retalho_cm > 20 → criar automaticamente produto "Retalho [produto] [comp]cm".

  REGRA 10 — DEVOLUÇÕES
    Produto retorna como DISPONIVEL, DEFEITUOSO ou DESCARTE.
    Crédito gerado em creditos_cliente com saldo_restante controlado.
    Crédito pode abater total em novas vendas.
    Movimentação DEVOLUCAO_CLIENTE obrigatória + log de auditoria.

  REGRA 11 — ORÇAMENTO → VENDA
    Ao converter: herda cliente, itens, desconto, observações.
    Orçamento → APROVADO referenciando a venda. Venda referencia orcamento_id.

  REGRA 12 — AVISOS DO CHEFE BLOQUEANTES
    URGENTE → Modal fullscreen, z-index:9999, sem fechar, só "Li e Entendi".
              Grava em leituras_aviso: aviso_id, funcionario_id, timestamp, IP, via.
              Usuário não acessa nenhuma tela antes de confirmar.
    NORMAL  → Modal fechável, mesma gravação.
    Interceptar IMEDIATAMENTE após login no frontend (React Router loader ou useEffect em App.tsx).
    Se funcionário logado receber aviso URGENTE via WebSocket: abrir modal imediatamente.

  REGRA 13 — LOGS DE AUDITORIA APPEND-ONLY
    NUNCA deletar registros de logs_auditoria. Exportável em CSV.

  REGRA 14 — BUSCA DE CEP (ViaCEP)
    Hook useCep.ts: ao digitar CEP válido (8 dígitos), chama ViaCEP e preenche campos.
    fetch(`https://viacep.com.br/ws/${cep}/json/`)
    Gratuito, sem chave de API, obrigatório no formulário de clientes.

  REGRA 15 — MULTIEMPRESA BASE
    Todo modelo tem empresa_id. Toda query inclui WHERE empresa_id = req.user.empresa_id.
    Nunca cruzar dados entre empresas.

  REGRA 16 — NF-e
    Homologação SEFAZ por padrão. Nunca produção sem config explícita.
    NUNCA reprocessar NF-e emitida — sempre cancelar e reemitir.

  REGRA 17 — STATUS DE MATERIAL (TRANSIÇÕES PERMITIDAS)
    EM_ESTOQUE   → RESERVADO     : ao abrir OP (automático)
    EM_ESTOQUE   → DEFEITUOSO    : almoxarife registra dano (manual)
    RESERVADO    → EM_ESTOQUE    : ao cancelar OP (automático)
    RESERVADO    → EM_PRODUCAO   : operador confirma retirada (manual)
    EM_PRODUCAO  → DESCARTADO    : consumo final ao concluir OP (automático)
    EM_PRODUCAO  → EM_ESTOQUE    : material devolvido sem usar (manual)
    DEFEITUOSO   → DESCARTADO    : admin decide (manual)
    DEFEITUOSO   → EM_ESTOQUE    : admin decide reaproveitar (manual + aviso)
    QUALQUER     → DEVOLVIDO     : devolução de cliente (manual)
    DEVOLVIDO    → EM_ESTOQUE    : admin aprova retorno (manual)
    DEVOLVIDO    → DEFEITUOSO    : admin classifica (manual)
    DEVOLVIDO    → DESCARTADO    : admin descarta (manual)
    Backend bloqueia transições inválidas (retornar 400).

  // Axios instance com interceptors.
  // Request interceptor: anexa Authorization: Bearer {token} de authStore.
  // Response interceptor: se 401 → tenta refresh; se refresh falhar → logout.
  // Toda função retorna o tipo correto de ApiResponse<T>.

  import axios from 'axios'
  import { useAuthStore } from '@/store/authStore'

  export const api = axios.create({ baseURL: '/api/v1' })

  api.interceptors.request.use(config => {
    const token = useAuthStore.getState().token
    if (token) config.headers.Authorization = `Bearer ${token}`
    return config
  })

  api.interceptors.response.use(
    res => res,
    async error => {
      if (error.response?.status === 401) {
        // tentar refresh → se falhar → useAuthStore.getState().logout()
      }
      return Promise.reject(error)
    }
  )

  // Zustand store. Persistido em sessionStorage.
  import { create } from 'zustand'
  import { persist } from 'zustand/middleware'

  interface AuthState {
    token: string | null
    refreshToken: string | null
    usuario: Usuario | null
    demoMode: boolean
    login: (data: LoginResponse) => void
    logout: () => void
    hasPermissao: (perm: string) => boolean
    hasAnyPermissao: (...perms: string[]) => boolean
  }

  export const useAuthStore = create<AuthState>()(
    persist(
      (set, get) => ({
        // ... implementação completa
        hasPermissao: (perm) => get().usuario?.permissoes.includes(perm) ?? false,
        hasAnyPermissao: (...perms) => perms.some(p => get().hasPermissao(p)),
      }),
      { name: 'vidrato-auth', storage: sessionStorageAdapter }
    )
  )

  // Hook para React Hook Form. Ao digitar 8 dígitos válidos, busca ViaCEP.
  export function useCep(setValue: UseFormSetValue<any>) {
    return useCallback(async (cep: string) => {
      const clean = cep.replace(/\D/g, '')
      if (clean.length !== 8) return
      const res = await fetch(`https://viacep.com.br/ws/${clean}/json/`)
      const data = await res.json()
      if (!data.erro) {
        setValue('endereco', data.logradouro)
        setValue('bairro',   data.bairro)
        setValue('cidade',   data.localidade)
        setValue('estado',   data.uf)
      }
    }, [setValue])
  }

  // MUTATIONS — escrita de dados:
  export function useCriarVenda() {
    const qc = useQueryClient()
    return useMutation({
      mutationFn: (data: CriarVendaInput) =>
        api.post<ApiResponse<Venda>>('/vendas', data).then(r => r.data.dados),
      onSuccess: () => {
        qc.invalidateQueries({ queryKey: ['vendas'] })
        qc.invalidateQueries({ queryKey: ['produtos'] }) // atualiza estoque
        toast.success('Venda confirmada com sucesso!')
      },
      onError: (err: AxiosError<ApiError>) => {
        toast.error(err.response?.data?.erro ?? 'Erro ao criar venda')
      }
    })
  }
